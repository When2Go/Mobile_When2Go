# 핵심 설계 원칙

> 이 프로젝트에서 코드 구조에 관한 결정을 내릴 때 기준이 되는 원칙들.

---

## 1. 화면보다 출발 판단 로직이 핵심

"지금 나가?"는 경로 안내 앱이 아니라 **출발 타이밍 판단 앱**이다. UI는 그 결과를 전달하는 수단. UI 화려함보다 계산 결과의 정확성과 알림 신뢰성이 먼저다.

→ 출발 시간 계산(`src/utils/departure.ts`), 알림 스케줄링(`src/utils/notification.ts`)은 로직 복잡도를 용납한다. 이 코드에는 단위 테스트를 우선 작성한다.

## 2. 서버 상태는 최대한 단순하게

MVP 규모에서 TanStack Query 전면 도입은 오버엔지니어링이다. 기본은 `useState + useEffect`, 폴링과 캐싱이 실제로 필요한 곳(GPS 트래킹, 즐겨찾기 목록)에만 TanStack Query 도입.

→ 복잡한 상태 관리가 필요하다고 느껴지면, 실제로 그런지 먼저 팀에 공유한다.

## 3. 디자인 시스템은 단일 소스

색상은 `src/constants/colors.ts` 또는 `tailwind.config.js` 시맨틱 토큰에서만 가져온다. 하드코딩 색상값은 코드 어디에도 존재해서는 안 된다.

→ 디자인 변경 = 토큰 파일 하나만 수정하면 전체 반영.

## 4. 공용 컴포넌트는 설계 단계에서 결정

"나중에 리팩토링"하는 공용화는 대부분 실제로 일어나지 않는다. 설계 단계에서 공용으로 쓰일 것이 확실한 컴포넌트(BottomSheetModal, AdBanner, MapView 래퍼 등)는 처음부터 `components/common/`에 만든다.

반대로, 화면 전용임이 명확한 컴포넌트는 공통화를 서두르지 않는다.

## 5. 중복 허용 > 성급한 결합

비슷한 코드가 2군데에 있다고 해서 바로 공통화하지 않는다. 미래에 다르게 변할 코드를 공통화하면 공통 컴포넌트가 분기로 가득 차게 된다. 3번 이상 동일 변경이 일어난 후 공통화를 검토한다.

## 6. 백엔드 경계 존중

ODsay API, 재계산 스케줄링, 알림 발송은 Spring Boot 백엔드 담당. 프론트는 결과를 받아서 표시하는 역할. 경계를 넘는 코드는 작성하지 않는다.
