# PR 본문 작성 가이드

> 모든 PR 본문은 **비전문가도 한 번 읽고 무슨 변경인지 이해할 수 있는 수준**으로 쓴다.
> 적용 대상: 사람이 직접 쓰는 PR, `github-issue-work` 스킬이 자동 생성하는 PR 모두.

---

## 원칙

1. **한 줄 요약을 가장 먼저**. 이 PR을 한 마디로 표현하면? 부터 시작.
2. **무엇을 / 왜 / 어떻게 쓰는지** 순서로. 라이브러리 이름 나열이 아니라 실제 동작 위주.
3. **고유명사·기술 용어는 풀어 쓰기**. 어쩔 수 없는 핵심 용어(예: NativeWind, BottomSheet)만 남기고, 그조차 한 줄 풀이를 옆에 붙인다.
4. **표·체크박스로 한눈에 보이게**. 자동 리뷰 결과, 테스트 체크리스트는 표로.
5. **직접 확인하는 법 섹션 필수**. 리뷰어가 어떤 명령으로 어떻게 검증할지 적는다.
6. **리뷰어 참고는 마지막에**. 머지 후 영향 범위, 의도적으로 건드리지 않은 영역 등.

---

## 안 좋은 예 vs 좋은 예

### ❌ 안 좋은 예 (라이브러리 나열 + 기술 용어 폭격)

```
- NativeWind v4 + tailwindcss 3.4 셋업 (babel/metro/tailwind config)
- ThemeContext: system/light/dark 3-mode + nativewind colorScheme 동기화
- MobileLayout (children pattern): 헤더 + 바텀탭 + 다크모드
- BottomSheetModal: @gorhom/bottom-sheet 래퍼
```

→ 무엇을 쓰는지는 적혀 있지만, **왜 필요한지·어떻게 쓰는지**가 없고 비전문가는 한 줄도 이해 못 한다.

### ✅ 좋은 예 (목적 + 사용법)

```
### 1. 스타일을 쓸 수 있게 셋업 (NativeWind)
- 지금까지는 RN의 style={{ ... }}로만 스타일을 줄 수 있었는데, 이제 웹처럼
  className="bg-blue-600 text-white" 같이 쓸 수 있다.
- 필요한 설정 파일을 다 만들었다.

### 5. 바텀시트 (BottomSheetModal)
- 아래에서 슥 올라오는 시트. 외부 라이브러리를 직접 쓰지 않고 우리 래퍼만 쓰도록 통일.
- 사용법: <BottomSheetModal isOpen={...} onClose={...} title="...">
```

→ 무엇을 / 왜 / 어떻게 쓰는지가 한 묶음.

## 권장 본문 구조

```markdown
## 관련 이슈
closes #N

## 한 줄 요약
이 PR이 무엇을 하는지를 한 문장으로.

## 무엇이 바뀌었나
### 1. {평이한 한국어 제목} ({필요시 핵심 용어})
- 무엇: ...
- 왜: ... (생략 가능 — 의도가 자명하면)
- 어떻게 쓰는지: 코드 한 줄 예시 또는 사용 시나리오

### 2. ...

## 직접 확인하는 법
```bash
npm install
npx expo start
```
- [ ] 확인 항목 1
- [ ] 확인 항목 2

## 자동 검사 결과 (when2go-qa)
| 검사 항목 | 결과 |
|---|---|
| 1. 타입·린트 에러 | ✅ 통과 |
| 2. 코드 품질 | ✅ 위반 0건 |
| 3. 디자인 시스템 | ✅ 위반 0건 |
| ... | ... |

**[CRITICAL] N건 / [WARNING] N건 / [INFO] N건** — 머지 가능 여부 한 줄 판정.

### INFO 상세 (머지 영향 없음)
1. ...

## 리뷰어 참고
- 의도적으로 건드리지 않은 영역
- 머지 후 영향 범위
- exec-plan 정리 시점
```

---

## 자동 리뷰 결과 첨부 규칙

- **반드시 표로 정리**. 카테고리별 ✅/❌ 한 줄.
- **[CRITICAL]/[WARNING]/[INFO] 건수**는 굵게 한 줄로.
- INFO는 별도 하위 섹션으로 풀어 쓰되, "머지 영향 없음" 라벨 명시.

---

## 톤 체크리스트 (PR 작성 후 자가 점검)

- [ ] 한 줄 요약을 읽고 동료가 \"아, 이런 작업이구나\"라고 이해할 수 있다
- [ ] 변경 사항에서 **라이브러리 이름 나열만 있는 줄**이 없다 (있다면 한 줄 설명 추가)
- [ ] 직접 확인하는 법이 있다
- [ ] 자동 리뷰 결과가 표로 정리돼 있다
- [ ] 비기술자 동료가 읽어도 50% 이상은 이해할 수 있다
